# Adapters — running vs-token-safer under other agents

vs-token-safer is **one engine** (`server/` — `runTool` + the language-server / tree-sitter / concept tiers)
behind **two universal transports**: a **CLI** (`vts`) and an **MCP server** (`server/index.js`). Both call
the same `runTool`, so the *answer* — a token-capped `file:line` list, no bodies, labeled with its precision
— is identical on every host. These adapters are **thin**: config snippets + routing docs that point a given
agent at that one engine. **No code fork** — a separate engine per harness would betray the charter (official
engines, one discipline). One repo, one version (the `bump.mjs` + marketplace parity guards keep it single).

## What ports, and what doesn't

| Pillar | Travels? | How |
| --- | --- | --- |
| **Capped** output (`file:line`, no source) | ✅ free | it's in the tool result |
| **Honest precision** (the completeness certificate, steers) | ✅ free | also in the tool result |
| **Enforced** swap (grep → vts) | ⚠️ degrades | only Claude Code has the rewrite hook |

So the tools work everywhere via MCP/CLI; only the *enforcement* needs a per-harness story.

## The enforcement ladder (graceful degradation)

1. **intercepted** — Claude Code's PreToolUse hook rewrites a stray code-`grep` into the equivalent `vts`
   call. Strongest: it even catches a grep the model would never have routed. (The `.claude-plugin/` + `hooks/`
   in the repo root.)
2. **instructed** — a routing block in the host's standing-instructions file (`AGENTS.md`, `.cursorrules`, …)
   tells the agent to prefer the vts tools by habit. Generated by `vts routing` so it stays in sync.
3. **available** — the raw MCP tools plus the steers baked into every result. If the model calls vts at all,
   it gets nudged toward the right rung; the only gap is a grep it never routed through vts.
